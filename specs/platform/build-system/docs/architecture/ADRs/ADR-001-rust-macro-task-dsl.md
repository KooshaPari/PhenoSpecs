# ADR-001: Rust Macro-Based Task Definition DSL

**Status**: Accepted  
**Date**: 2026-04-04  
**Author**: phenoForge Core Team  
**Reviewers**: Architecture Review Board  

---

## Context

phenoForge requires a mechanism for users to define tasks. We evaluated multiple approaches:

1. **Configuration files** (TOML/YAML) - Used by Task, Nx
2. **Domain-specific language** - Used by Bazel (Starlark)
3. **Shell scripts** - Used by Just, Make
4. **Native code with macros** - Used by Mage (Go)

Each approach has tradeoffs in ergonomics, performance, type safety, and tooling support.

## Decision

We will use Rust procedural macros to create a task definition DSL embedded in Rust source code.

### Implementation

```rust
// Task definition using macros
#[task]
#[deps(build)]
#[cache]
fn test() -> TaskResult {
    sh!("cargo test --all-features")
        .env("RUST_LOG", "debug")
        .run()
}
```

The macro system will provide:
- `#[task]` - Marks a function as a task
- `#[deps(...)]` - Declares task dependencies
- `#[cache]` - Enables output caching
- `#[env(...)]` - Sets environment variables
- `#[remote]` - Allows remote execution

## Rationale

### Why Native Rust Code

1. **Type Safety**: Compile-time error detection
   ```rust
   // This will fail at compile time if 'build' doesn't exist
   #[deps(build)]
   fn test() {}
   ```

2. **IDE Support**: Standard Rust tooling (rust-analyzer, IntelliJ)
   - Autocomplete for task names
   - Go-to-definition
   - Refactoring support

3. **Performance**: No parsing overhead at runtime
   - Task graph built at compile time
   - Zero-cost abstractions

4. **Ecosystem**: Full access to Rust crates
   - Any library can be used in tasks
   - Async/await native support

5. **Version Control**: Tasks in Git, not external files
   - Atomic commits with task changes
   - Code review for build logic

### Why Not Alternatives

| Approach | Reason for Rejection |
|----------|---------------------|
| YAML/JSON | Runtime errors, no IDE support, verbose |
| Starlark-like DSL | Requires custom parser, learning curve, limited tooling |
| Shell scripts | No type safety, platform differences, escape nightmares |
| External DSL | Fragmentation, context switching, limited expressiveness |

### Comparison with Mage (Go)

Mage proved this approach works at scale. We improve upon it by:
- Using Rust's more expressive macro system
- Built-in async support via Tokio
- Better dependency declaration (attribute vs function call)
- Caching and remote execution as first-class

## Consequences

### Positive

1. **Developer Experience**:
   - Type-safe task definitions
   - Excellent error messages
   - Full IDE support

2. **Performance**:
   - <50ms cold start
   - No runtime parsing
   - Efficient dependency resolution

3. **Correctness**:
   - Compile-time dependency validation
   - No missing dependency surprises
   - Refactoring safety

4. **Integration**:
   - Seamless Cargo workspace support
   - Easy integration with existing Rust projects
   - Phenotype ecosystem integration

### Negative

1. **Language Barrier**: Requires Rust knowledge for task authors
   - Mitigation: Provide template tasks
   - Mitigation: Script task type for shell commands

2. **Compilation Time**: Tasks must compile before running
   - Mitigation: Fast check mode (parse only)
   - Mitigation: Background compilation in watch mode

3. **Learning Curve**: Rust macros can be complex
   - Mitigation: Simple, documented API
   - Mitigation: Examples and templates

### Neutral

1. **Syntax**: May look unfamiliar to Make/Task users
   - But consistent with Cargo.toml and other Rust tools

## Implementation Details

### Macro Expansion

```rust
// Input
#[task]
#[deps(build)]
fn test() -> TaskResult {
    sh!("cargo test").run()
}

// Expanded
const __TASK_TEST: TaskDef = TaskDef {
    name: "test",
    dependencies: &["build"],
    handler: || { sh!("cargo test").run() },
    cache_enabled: false,
    remote_enabled: false,
};
```

### Registration System

```rust
// Generated by macro
inventory::submit! {
    TaskRegistration {
        name: "test",
        def: &__TASK_TEST,
    }
}
```

## Alternatives Considered

### Alternative 1: TOML Configuration

```toml
[[task]]
name = "test"
deps = ["build"]
cmd = "cargo test"
```

**Rejection**: Runtime errors, no type safety, poor IDE support.

### Alternative 2: Just-like Syntax

```justfile
test: build
    cargo test
```

**Rejection**: Requires custom parser, no compile-time checks, limited expressiveness.

### Alternative 3: Starlark (Bazel-style)

```python
test_task = task(
    name = "test",
    deps = ["build"],
    cmd = "cargo test",
)
```

**Rejection**: New language to learn, JVM dependency, complex.
