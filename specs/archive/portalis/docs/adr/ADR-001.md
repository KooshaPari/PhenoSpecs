# ADR-001: Technology Stack Selection

**Status**: Accepted

**Date**: 2026-04-04

**Context**: Portalis requires a technology stack that delivers high performance (50,000+ req/s, <5ms P99 latency), memory efficiency (<1KB/request), and operational simplicity. We evaluated multiple languages and frameworks for building the API gateway.

---

## Decision Drivers

| Driver | Priority | Notes |
|--------|----------|-------|
| Performance | High | Must handle 50K+ req/s with minimal latency |
| Latency consistency | High | No GC pauses affecting P99 latency |
| Memory efficiency | High | Cost optimization at scale |
| Developer productivity | Medium | Reasonable iteration speed |
| Operational simplicity | Medium | Single binary, minimal dependencies |
| Ecosystem | Medium | Available libraries for HTTP, TLS, metrics |

---

## Options Considered

### Option 1: Rust with Tokio

**Description**: Build the gateway in Rust using Tokio for async runtime, Hyper for HTTP, and Tower for middleware.

**Pros**:
- Highest performance with zero-cost abstractions
- No garbage collection pauses - deterministic latency
- Memory safety without GC overhead
- Excellent async ecosystem (Tokio, Hyper, Tower)
- Single binary deployment
- Cloudflare's Pingora proves Rust works at scale

**Cons**:
- Steeper learning curve for async Rust
- Longer compilation times
- Smaller middleware ecosystem than Node.js/Go

**Performance Data**:
| Metric | Value | Source |
|--------|-------|--------|
| Throughput | 50,000+ req/s | Benchmark projections |
| P99 Latency | <5ms | Projections based on Pingora |
| Memory/request | <0.5KB | Memory safety guarantees |
| GC pauses | None | Rust ownership model |

### Option 2: Go with Gin/Echo

**Description**: Build the gateway in Go using Gin or Echo framework.

**Pros**:
- Good performance with Goroutines
- Simpler async model than Rust
- Faster compilation
- Large ecosystem for networking
- Easy to hire Go developers

**Cons**:
- Runtime GC causes latency spikes at high percentiles
- Larger memory footprint than Rust
- GC pauses affect P99 latency

**Performance Data**:
| Metric | Value | Source |
|--------|-------|--------|
| Throughput | 15,000-20,000 req/s | Tyk benchmarks |
| P99 Latency | 8-10ms | Industry data |
| Memory/request | 1-1.5KB | Industry data |
| GC pauses | 1-5ms | Go runtime behavior |

### Option 3: C++ with Envoy

**Description**: Use Envoy proxy as the base with custom extensions.

**Pros**:
- Highest raw performance
- Battle-tested at scale (used by Lyft, Google)
- Rich feature set including service mesh

**Cons**:
- Complex xDS configuration model
- Steep learning curve
- Difficult to extend with custom logic
- Heavy for simple API gateway use cases

**Performance Data**:
| Metric | Value | Source |
|--------|-------|--------|
| Throughput | 20,000 req/s | Envoy benchmarks |
| P99 Latency | 5ms | Envoy documentation |
| Memory/request | 1KB | Envoy documentation |
| Config complexity | High | xDS protocol |

### Option 4: Node.js with Fastify

**Description**: Build the gateway in Node.js using Fastify framework.

**Pros**:
- Fastest development iteration
- Largest ecosystem for web libraries
- Easy to add features with npm packages
- Familiar to most developers

**Cons**:
- GC pauses at high request rates
- Lower throughput than compiled languages
- Not suitable for high-performance scenarios
- Higher memory usage

**Performance Data**:
| Metric | Value | Source |
|--------|-------|--------|
| Throughput | 5,000-10,000 req/s | Fastify benchmarks |
| P99 Latency | 15-20ms | Industry data |
| Memory/request | 2-4KB | Node.js runtime |
| GC impact | Significant | V8 GC behavior |

---

## Decision

**Chosen Option**: Rust with Tokio

**Rationale**: Rust provides the best combination of performance, latency consistency, and memory efficiency. The absence of garbage collection eliminates latency spikes at high percentiles, which is critical for an API gateway. While the learning curve is steeper, the long-term benefits of deterministic performance outweigh the initial investment.

The Tokio ecosystem provides all necessary components (Hyper for HTTP, Tower for middleware, Tokio for async runtime) that have been proven in production at companies like Cloudflare (Pingora) and Discord.

**Evidence**:
- Cloudflare's migration from NGINX to Pingora (Rust) reduced CPU usage by 70%
- Discord's use of Rust for performance-critical services
- Kong's performance limitations with Lua/NGINX
- Industry benchmarks showing Rust outperforming Go by 2-3x for API workloads

## Performance Benchmarks

```bash
# Benchmark command for Rust gateway
wrk -t12 -c400 -d30s --latency http://localhost:8080/health

# Expected results for Rust implementation:
# - P50: <1ms
# - P99: <5ms
# - P999: <10ms
# - Throughput: 50,000+ req/s

# Comparison benchmark with Go
# Install Go and Gin for comparison
go install github.com/gin-gonic/gin@latest

# Kong benchmark (requires Kong installation)
# docker run -d --name kong kong:latest
# wrk -t12 -c400 -d30s --latency http://localhost:8000/health
```

**Results**:

| Benchmark | Rust (Projected) | Go (Tyk reference) | Improvement |
|-----------|------------------|------------------|-------------|
| P50 Latency | 0.8ms | 3ms | 73% faster |
| P99 Latency | 3ms | 8ms | 62.5% faster |
| P999 Latency | 8ms | 25ms | 68% faster |
| Throughput | 50,000 req/s | 15,000 req/s | 3.3x higher |
| Memory/req | 0.5KB | 1.5KB | 67% lower |

## Consequences

### Positive

- Superior performance with deterministic latency
- Memory efficiency reduces infrastructure costs
- Single binary simplifies deployment
- No runtime dependencies
- Strong type safety reduces bugs

### Negative

- Steeper learning curve for async Rust
- Longer compilation times (mitigated by caching)
- Smaller ecosystem requires more custom code
- Recruiting harder than Go/Node.js developers

### Neutral

- Team skill development required
- Some features may take longer to implement initially
- Community support smaller than Go/Node.js
